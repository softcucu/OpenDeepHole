# 高风险模块识别

## 目标

分析代码仓中的目录、文件、类、函数、接口、配置和调用关系，识别高风险模块，并判断各模块涉及的高风险代码类型及外部暴露情况。

只进行高风险模块识别，不分析具体漏洞，不输出风险等级、攻击路径或整改建议。

## 输入

输入可以是：

- 代码仓根目录
- 指定目录或文件
- 目录结构
- 源代码
- 配置文件
- 接口定义
- 项目文档

## 识别流程

1. 查看代码仓目录结构、主要入口和对外接口。
2. 定位管理控制、协议解析、外部数据处理、安全功能、敏感数据处理和 Web 相关代码。
3. 结合类、函数、接口、路由和调用关系判断模块实际功能。
4. 将功能相同或关联紧密的代码合并为一个模块。
5. 判断模块面临的主要威胁及各项高风险特征。
6. 按规定字段输出高风险模块列表。

目录名、文件名和关键词只能作为识别线索，应结合代码实际功能进行判断。

## 识别规则

### 1. 管理和控制接口相关代码

符合以下任一情况时标记为 `是`：

- 对外提供设备或系统管理接口
- 提供配置修改、命令执行或状态控制能力
- 能够执行启动、停止、重启、升级或复位
- 实现 Telnet、FTP、MML、CLI、SNMP 等管理接口
- 实现路由协议、信令接口或控制面协议
- 接收远程管理或控制指令

仅包含普通静态配置的代码不属于此类。

### 2. 不可信来源数据解析或处理代码

符合以下任一情况时标记为 `是`：

- 解析网络报文、消息、协议或用户输入
- 解析上传文件、导入文件或外部文档
- 执行反序列化、解码或格式转换
- 处理用户面数据、消息队列或设备上报数据
- 根据外部输入执行计费策略、规则或业务逻辑
- 处理来自第三方系统的数据

不可信来源包括网络请求、用户输入、上传文件、外部系统、消息队列、设备上报数据及外部用户可写入的数据。

### 3. 安全相关类代码

符合以下任一情况时标记为 `是`：

- 身份认证或设备认证
- 授权、权限检查或接入控制
- 会话、令牌或凭据管理
- 加密、解密、签名或验签
- 密钥、证书或密码管理
- 安全日志或日志审计
- 软件完整性校验
- 软件包签名验证
- 安全策略执行

普通数据去重或非安全用途的哈希代码不属于此类。

### 4. 个人数据或者敏感数据代码

符合以下任一情况时标记为 `是`：

- 集中存储、查询、导入、导出或批量处理个人数据
- 对个人数据进行脱敏、匿名化或去标识化
- 集中处理密码、密钥、令牌等敏感数据
- 集中处理身份、位置、通信、支付或用户行为数据
- 对个人数据或敏感数据执行加密、解密或访问控制

个人数据或敏感数据包括姓名、身份证件号码、电话号码、邮箱、地址、位置数据、账号标识、设备标识、生物特征、通信记录、支付信息、密码、密钥和访问令牌等。

### 5. Web 相关处理

符合以下任一情况时标记为 `是`：

- 处理 HTTP 请求、响应、请求头或响应头
- 实现 Web API、Controller、Endpoint 或路由
- 处理 Cookie、Session 或 Web Token
- 处理文件或图片上传、下载
- 构造或执行 SQL、LDAP 查询
- 解析或生成 XML
- 使用 XPath、XSLT 等 XML 处理能力
- 生成、渲染或输出 HTML
- 处理 URL 参数、表单或请求体

### 6. 外部暴露面

模块存在以下任一情况时，`是否外部暴露面` 标记为 `是`：

- 可通过网络、Web、协议、消息、文件上传或远程命令直接访问
- 接收来自用户、设备、第三方系统或其他不可信来源的输入
- 对外提供 API、端口、服务、管理接口、控制接口或协议接口
- 位于外部输入进入系统后的首个或主要处理环节

仅被内部可信模块调用，且不直接接收外部输入或提供对外接口时，标记为 `否`。

## 面临威胁填写规则

`面临威胁` 应根据模块功能填写简明的潜在威胁，例如：

- 未授权访问或越权操作
- 管理命令滥用
- 配置篡改
- 恶意报文或畸形输入攻击
- 文件解析攻击
- 注入攻击
- 身份伪造或认证绕过
- 密钥或凭据泄露
- 敏感数据泄露或篡改
- 恶意文件上传
- 跨站脚本或响应内容注入
- 服务拒绝

只描述模块可能面临的威胁类型，不判断代码中是否已经存在对应漏洞。

## 代码目录填写规则

`代码目录` 应填写该高风险模块在代码仓中的主要相对目录或文件路径。

要求：

- 优先填写目录路径，例如 `src/auth`、`services/device/config`。
- 如果模块集中在单个文件中，可以填写文件路径，例如 `src/auth/login.go`。
- 如果模块由多个目录共同构成，使用字符串数组，列出主要目录或文件。
- 路径应来自代码仓，不得填写虚构路径。
- 不要用代码目录代替模块名称；模块名称仍应是业务或功能名称。

## 判断原因要求

`判断为高风险模块的原因` 必须结合代码分析证据详细说明为什么该模块属于高风险模块。

该字段应至少说明：

- 代码仓库中哪些目录、文件、类、函数、接口、路由、配置或调用关系体现了该模块职责；
- 该模块命中了哪些高风险代码类型，为什么对应字段应填写 `是`；
- 该模块是否直接接收外部输入、提供外部接口或处于外部输入主要处理环节，为什么 `是否外部暴露面` 应为 `是` 或 `否`；
- 为什么这些文件、接口或功能应合并为同一个业务模块，而不是逐文件罗列；
- 模块面临威胁与模块职责之间的关系。

原因说明应具体到可核验的代码或文档依据。可以引用相对路径、类名、函数名、接口名、路由、配置项或调用关系。不得只写“存在相关代码”“该模块风险较高”等泛化表述。

## 输出要求

仅输出 JSON，不得输出 Markdown 表格、解释文字、代码块外说明或其他内容。

输出必须是合法的 JSON 数组。数组中的每个对象表示一个高风险模块。

每个对象必须且仅包含以下 10 个字段，字段名称不得修改、增加或删除：

1. `模块名称`
2. `代码目录`
3. `面临威胁`
4. `是否涉及设备或系统对外提供管理和控制接口相关的代码`
5. `是否涉及对不可信来源数据进行解析或处理的代码`
6. `是否涉及安全相关类代码(如，认证、授权、接入控制、加解密、密钥管理、日志审计、软件完整性保护等模块)`
7. `是否涉及个人数据或者敏感数据的代码`
8. `是否涉及web相关处理`
9. `是否外部暴露面`
10. `判断为高风险模块的原因`

约束：

- 所有“是否”字段的值只能是字符串 `是` 或 `否`。
- `模块名称` 必须使用代码仓中的实际业务或功能名称。
- `代码目录` 必须是字符串或字符串数组，填写代码仓中的相对目录或文件路径。
- `面临威胁` 使用字符串填写，可包含多个威胁，使用中文顿号或逗号分隔。
- `判断为高风险模块的原因` 必须结合代码分析证据详细说明模块职责、高风险特征、外部暴露判断和威胁依据。
- 同一模块可以有多个“是否”字段标记为 `是`。
- 不逐文件机械罗列，应将功能相同或关联紧密的代码合并为业务模块。
- 不得虚构代码仓中不存在的模块、功能或代码目录。
- 不得输出模块描述、风险类别、置信度、漏洞信息、风险等级、攻击路径或整改建议。
- 没有识别出高风险模块时，输出空数组 `[]`。
- 必须保证输出可被标准 JSON 解析器直接解析。
- 不得在 JSON 中添加注释。
- 不得使用尾随逗号。
