# 攻击树分析

## 1. 目标

读取价值资产与高风险模块文档、攻击模式参考文档，并结合对应代码库进行分析，从外部攻击者视角构建攻击树。默认攻击模式参考文档为本 skill 内置的 `references/attack_mode.json`。

攻击树用于描述以下理论攻击路径：

`外部暴露的高风险模块 → 内部模块 → 价值资产攻击目标`

每条攻击路径还需要：

- 从 `references/attack_mode.json` 中选择与该路径相匹配的攻击模式
- 列出该路径关联的全部高风险模块
- 区分外部暴露的高风险模块和非外部暴露的高风险模块

攻击树仅用于分析模块之间可能存在的攻击影响关系，不用于确认具体漏洞是否存在，也不用于判断漏洞是否可以被实际利用。

## 2. 输入

输入包括：

1. 价值资产与高风险模块 Markdown 文档
2. 攻击模式参考文档，默认使用本 skill 的 `references/attack_mode.json`
3. 与上述文档对应的代码库

### 2.1 价值资产

每个价值资产至少包含：

- 资产名
- 资产类别
- 资产描述
- 攻击损失

资产类别有且仅有：

- 数据资产
- 软件资产
- 硬件资产
- 服务资产

### 2.2 高风险模块

每个高风险模块包含：

- 模块名称
- 面临威胁
- 是否涉及设备或系统对外提供管理和控制接口相关的代码
- 是否涉及对不可信来源数据进行解析或处理的代码
- 是否涉及安全相关类代码
- 是否涉及个人数据或者敏感数据的代码
- 是否涉及 Web 相关处理
- 是否外部暴露面

高风险模块既可能是外部暴露模块，也可能是非外部暴露的内部模块。

### 2.3 攻击模式参考文档

攻击模式参考文档提供可供选择的攻击模式。

执行本 skill 时，必须读取 `references/attack_mode.json` 作为默认攻击模式参考。该文件是 JSON 数组，每个对象通常包含：

- 攻击模式编号
- 攻击模式标签
- 攻击模式名称
- 攻击模式描述

分析时只能从 `references/attack_mode.json` 中选择攻击模式，不得自行创建、改写或补充该文件中不存在的攻击模式。

当 `attack_mode.json` 未提供某个字段时，仅使用其实际提供的信息，不得推测补全。

### 2.4 代码库

代码库用于分析：

- 模块划分
- 高风险模块对应的代码实现
- 高风险模块的外部暴露关系
- 模块之间的调用关系
- 模块之间的数据传递关系
- 模块之间的控制或依赖关系
- 模块与价值资产之间的关系

代码库分析只用于支持理论上的模块影响路径，不用于确认漏洞存在。

## 3. 分析原则

### 3.1 理论分析原则

攻击树分析是基于系统架构、模块职责、代码调用关系、数据流关系和依赖关系进行的理论威胁推演。

分析过程中：

- 不确认具体漏洞是否存在
- 不进行漏洞扫描
- 不进行漏洞利用验证
- 不判断具体漏洞是否可利用
- 不将缓冲区溢出、命令注入、认证绕过、越权等具体漏洞作为攻击树节点
- 不因代码中存在某类危险函数，就直接认定系统存在对应漏洞
- 可以根据模块职责和模块关系，分析某个模块受到攻击后可能影响哪些内部模块和价值资产

### 3.2 模块关系原则

代码库用于确认：

- 哪些高风险模块具有外部暴露面
- 外部暴露高风险模块能够调用或影响哪些内部模块
- 内部模块之间是否存在调用、数据传递、控制或依赖关系
- 哪些模块能够访问、处理、控制或保护价值资产
- 攻击路径中经过哪些高风险模块

模块之间的连接表示理论上的影响传播关系，不表示已经确认存在漏洞利用链。

### 3.3 攻击模式关联原则

攻击模式用于描述一条攻击路径所体现的通用攻击方式或影响方式。

攻击模式的选择必须满足：

- 攻击模式来自 `references/attack_mode.json`
- 攻击模式描述与攻击路径中的入口类型、模块职责、数据流、控制关系或影响目标相匹配
- 关联依据来自价值资产、高风险模块、代码库和 `references/attack_mode.json`
- 不以确认某个具体漏洞存在作为匹配前提
- 不得仅因名称相似就进行关联
- 同一攻击路径可以关联一个或多个攻击模式
- 同一个攻击模式可以关联多条攻击路径
- 无法合理匹配时，攻击模式列表输出空数组，并在分析缺口中说明

## 4. 节点定义

### 4.1 根节点

根节点表示外部攻击者针对价值资产的最终攻击目标。

每个价值资产至少对应一棵攻击树。根节点名称格式为：

`攻击价值资产：<资产名>`

根节点的攻击目标应结合资产类别、资产描述、攻击损失以及代码中的实际功能确定，例如：

- 数据资产：导致数据泄露、篡改、删除或不可用
- 软件资产：导致软件逻辑、配置、策略或完整性受到影响
- 硬件资产：导致硬件被非法控制、改变状态、停用或破坏
- 服务资产：导致服务被未授权使用、劫持、降级或中断

根节点必须对应输入中的价值资产，不得将具体漏洞、外部接口或内部模块作为根节点。

### 4.2 内部节点

内部节点表示攻击路径中的内部模块。

内部模块是指：

- 不直接对外暴露的业务模块、功能模块、数据处理模块、安全模块或基础能力模块
- 能够被外部暴露的高风险模块直接或间接调用、影响或传递不可信数据
- 能够继续调用或影响其他内部模块
- 能够访问、处理、控制或保护价值资产
- 位于外部暴露高风险模块与价值资产之间

内部节点既可以是普通内部模块，也可以是非外部暴露的高风险模块。

当内部模块属于高风险模块列表时，必须在 JSON 中标记：

- `is_high_risk_module` 为 `true`
- `external_exposure` 为 `false`

内部节点名称必须是代码库中能够识别的具体模块、组件、子系统或具有明确模块职责的功能单元。

以下内容不得作为内部节点：

- 触发缓冲区越界读写
- 利用命令注入漏洞
- 绕过身份认证
- 获取管理员权限
- 构造恶意报文
- 发送异常请求
- 造成服务中断
- 导致数据泄露
- 其他漏洞名称、攻击动作、攻击条件或攻击结果

### 4.3 叶子节点

叶子节点表示外部暴露的高风险模块，是攻击树的外部攻击入口。

叶子节点必须同时满足：

- 属于输入文档中的高风险模块
- “是否外部暴露面”为是，或者能够结合代码库确认其对外提供接口或接收外部不可信输入
- 外部攻击者能够直接向该模块提供请求、报文、文件、消息、命令或其他输入
- 该模块能够直接或间接影响内部模块或价值资产

叶子节点必须使用高风险模块的模块名称。

以下内容不得作为叶子节点：

- 具体接口路径
- 端口
- 函数
- 参数
- 漏洞名称
- 攻击动作
- 权限状态
- 不对外暴露的内部模块

具体接口、端口或输入形式可以作为叶子节点的说明信息，但不能代替外部暴露高风险模块成为节点。

## 5. 节点关系

节点之间的连接表示模块间的理论攻击影响关系。

连接方向为：

`叶子节点 → 内部节点 → 根节点`

其含义包括：

- 叶子节点到内部节点：外部暴露高风险模块受到攻击后，可能通过调用、数据传递、控制关系或依赖关系影响内部模块
- 内部节点到内部节点：一个内部模块受到影响后，可能继续影响另一个内部模块
- 内部节点到根节点：内部模块受到影响后，可能影响价值资产并达成攻击目标
- 叶子节点到根节点：外部暴露高风险模块能够直接访问、处理、控制或保护价值资产，因此可能直接影响价值资产

所有能够成立的模块影响路径均独立输出，不分析或标记节点之间的逻辑关系。

## 6. 分析方法

### 步骤 1：解析输入

提取：

- 全部价值资产
- 全部高风险模块
- 全部攻击模式
- 代码库中的模块和模块关系

保留输入中的原始名称和描述。

### 步骤 2：确定根节点

针对每个价值资产建立独立攻击树。

结合资产类别、资产描述、攻击损失以及代码中的实际功能，确定对应的价值资产攻击目标。

### 步骤 3：确定叶子节点

从高风险模块列表中筛选外部暴露的高风险模块。

只有满足以下条件的高风险模块才能作为叶子节点：

- 输入文档标记“是否外部暴露面”为是，或者代码库能够确认其具有外部接口或外部输入入口
- 能够通过模块调用、数据流、控制关系或依赖关系影响内部模块或价值资产

### 步骤 4：识别内部节点

从代码库中识别位于叶子节点和价值资产之间的内部模块。

内部模块应满足至少一个条件：

- 被叶子节点对应的高风险模块调用
- 接收叶子节点传递的数据、消息、配置或控制指令
- 被其他内部模块调用或影响
- 能够访问、处理、控制或保护价值资产
- 在模块调用链或数据流中连接外部高风险模块和价值资产

识别内部节点时，检查该模块是否属于高风险模块列表。

非外部暴露的高风险模块可以作为内部节点，并需要保留其高风险模块属性。

### 步骤 5：建立模块影响路径

按照以下形式建立攻击路径：

`外部暴露高风险模块 → 内部模块 → 其他内部模块 → 价值资产攻击目标`

路径依据包括：

- 函数或模块调用关系
- 数据传递关系
- 消息、事件或任务传递关系
- 配置下发和加载关系
- 控制和依赖关系
- 对价值资产的访问、处理、修改、传输、保护或控制关系

攻击路径表示理论上的影响传播关系，不表示已经确认存在可利用漏洞。

每条能够独立从叶子节点到达根节点的模块影响路径均单独输出。

### 步骤 6：识别路径关联的高风险模块

针对每条攻击路径，按路径顺序检查其中的所有模块。

将以下模块加入该路径的 `related_high_risk_modules`：

- 作为叶子节点的外部暴露高风险模块
- 作为内部节点的非外部暴露高风险模块
- 能够直接影响价值资产且属于高风险模块列表的模块

不得把不在高风险模块输入文档中的模块标记为高风险模块。

每个关联高风险模块需要说明：

- 模块名称
- 在路径中的节点 ID
- 是否外部暴露
- 在路径中的角色
- 与该路径的关联说明

### 步骤 7：为路径匹配攻击模式

针对每条攻击路径：

1. 分析叶子节点的模块类型、外部输入形式和面临威胁
2. 分析路径中内部模块的职责和影响传播方式
3. 分析根节点对应的价值资产和攻击目标
4. 在 `references/attack_mode.json` 中查找适用场景相符的攻击模式
5. 选择能够描述该路径整体攻击方式或关键影响方式的攻击模式
6. 输出攻击模式编号、名称以及关联依据

不得：

- 输出参考文档中不存在的攻击模式
- 修改攻击模式编号或名称
- 仅根据关键词相同进行匹配
- 将具体漏洞是否存在作为攻击模式关联结论

### 步骤 8：检查完整性

检查：

- 每个根节点是否对应一个价值资产
- 每个叶子节点是否为外部暴露高风险模块
- 每个内部节点是否为代码库中的内部模块
- 非外部暴露高风险模块是否被正确标记为内部节点
- 每条路径是否从叶子节点到达根节点
- 每条路径是否列出全部关联高风险模块
- 每个攻击模式是否来自参考文档
- 攻击模式与路径是否存在合理匹配依据
- 是否错误地将漏洞、攻击动作、攻击条件或攻击结果作为节点
- 是否对具体漏洞是否存在作出结论
- 是否存在孤立节点、重复路径或循环路径

## 7. 输出要求

输出必须是合法 JSON，不得输出 Markdown 表格、Mermaid、解释性文字或 JSON 之外的其他内容。

### 7.2 JSON 设计说明

#### `nodes`

所有根节点、内部节点和叶子节点统一放入 `nodes` 数组，通过 `node_type` 区分。

这样可以：

- 避免分别维护多个节点数组
- 便于路径通过节点 ID 引用
- 便于标记内部节点是否属于高风险模块
- 便于统一校验节点唯一性

#### `edges`

`edges` 表示节点之间的模块影响关系。

每条边必须引用已经存在的节点 ID。

`influence_type` 有且仅有：

- `调用`
- `数据传递`
- `消息传递`
- `控制`
- `依赖`
- `直接影响`

#### `attack_paths`

每条攻击路径通过 `node_ids` 和 `edge_ids` 按顺序引用节点和边。

每条路径必须包含：

- 一个叶子节点
- 零个或多个内部节点
- 一个根节点
- 关联的全部高风险模块
- 从参考文档中选择的攻击模式

#### `related_high_risk_modules`

该字段必须列出攻击路径涉及的全部高风险模块，包括：

- 外部暴露高风险模块
- 非外部暴露高风险模块

`path_role` 有且仅有：

- `外部攻击入口`
- `内部影响模块`
- `直接资产影响模块`

同一个高风险模块在同一条路径中只输出一次。

#### `attack_patterns`

攻击模式必须来自 `references/attack_mode.json`。

每个攻击模式至少输出：

- `pattern_id`
- `pattern_name`
- `association_description`

`pattern_id` 必须来自 `attack_mode.json` 中的 `攻击模式编号`。当 `攻击模式编号` 是数组时：

- 数组只有一个编号时，输出该编号字符串
- 数组有多个编号时，按原顺序用中文顿号 `、` 连接为一个字符串

`pattern_name` 必须来自 `attack_mode.json` 中的 `攻击模式名称`。

如果攻击模式参考文档没有攻击模式编号：

- `pattern_id` 输出空字符串
- `pattern_name` 必须使用参考文档中的原始名称

如果一条路径无法匹配任何攻击模式：

- `attack_patterns` 输出空数组

### 7.3 JSON 字段约束

- `attack_trees`：攻击树列表，每个价值资产至少对应一棵攻击树
- `nodes`：该攻击树使用的全部节点
- `edges`：该攻击树中的全部模块影响关系
- `attack_paths`：从叶子节点到根节点的完整理论影响路径
- 每个 `node_id`、`edge_id` 和 `path_id` 必须唯一
- 根节点的 `module_name` 必须为 `null`
- 根节点的 `is_high_risk_module` 必须为 `false`
- 叶子节点的 `is_high_risk_module` 和 `external_exposure` 必须为 `true`
- 内部节点的 `external_exposure` 必须为 `false`
- 内部节点是否属于高风险模块，根据高风险模块输入文档确定
- 普通内部模块的 `is_high_risk_module` 为 `false`
- 非外部暴露高风险模块作为内部节点时，`is_high_risk_module` 为 `true`
- 外部高风险模块能够直接影响价值资产时，路径可以只包含叶子节点和根节点
- 路径中的第一个节点必须是叶子节点
- 路径中的最后一个节点必须是根节点
- `node_ids` 和 `edge_ids` 必须按照影响传播顺序排列
- `related_high_risk_modules` 必须覆盖路径中所有 `is_high_risk_module` 为 `true` 的节点
- `attack_patterns` 中的攻击模式必须来自 `references/attack_mode.json`
- 不得输出任何节点逻辑关系字段
- 不得输出注释
- 不得输出未定义字段
- 不得在 JSON 外输出任何文字

## 8. 强制约束

- 必须结合价值资产、高风险模块文档、`references/attack_mode.json` 和代码库进行分析
- 攻击树仅进行理论分析，不确认具体漏洞是否存在
- 不进行漏洞扫描、漏洞验证或漏洞利用分析
- 根节点必须是价值资产攻击目标
- 内部节点必须是内部模块
- 叶子节点必须是外部暴露的高风险模块
- 非外部暴露的高风险模块可以作为内部节点
- 每条攻击路径必须列出关联的全部高风险模块
- 每条攻击路径必须尝试从 `references/attack_mode.json` 中匹配攻击模式
- 不得虚构、改写或补充攻击模式
- 不得将具体接口、端口、函数、参数、漏洞、攻击动作、攻击条件或攻击结果作为叶子节点
- 不得将漏洞、攻击动作、攻击条件、权限状态或攻击结果作为内部节点
- 不得输出“触发缓冲区越界读写”之类的内部节点
- 不得虚构价值资产、高风险模块、内部模块或模块关系
- 输出中引用价值资产时必须使用本次输入价值资产的原始资产名、类别、描述和攻击损失；输出中引用高风险模块时必须使用高风险模块输入中的原始`模块名称`，不得改写为同义名称或子模块名称
- 普通内部节点可以是代码库中新识别的内部模块，不要求出现在高风险模块列表中；只有被标记为`is_high_risk_module: true`、作为叶子节点或出现在`related_high_risk_modules`中的模块，才必须来自高风险模块输入列表
- 模块之间的关系必须能够由输入文档、代码调用、数据流、控制关系、依赖关系或系统架构支撑
- 不分析、不判断、不标记任何节点逻辑关系
- 输入文档、`references/attack_mode.json` 或代码信息不足时，不得虚构无法支撑的路径、模块关系或攻击模式
- 输出必须是合法 JSON
- 输出不得包含 Markdown、Mermaid 或 JSON 之外的其他内容
