# 高风险模块合并

## 目标

读取多个 `high-risk-module-map` agent 输出的高风险模块候选，合并为最终高风险模块列表。

只做高风险模块合并，不分析具体漏洞，不输出风险等级、攻击路径或整改建议。

## 输入

输入通常包括：

- 多个高风险模块候选 JSON

候选对象通常包含：

- `模块名称`
- `代码目录`
- `面临威胁`
- 6 个“是否”字段
- `判断为高风险模块的原因`

## 合并规则

必须做语义合并，不能只按模块名称去重。

以下情况应合并为同一个业务模块：

- 名称不同，但代码目录、接口、类、函数、路由或配置指向同一功能模块。
- 一个候选描述入口层，另一个候选描述同一业务能力的服务层、处理层或数据层。
- 多个候选分别命中同一模块的不同高风险特征。
- 候选的判断原因显示它们职责紧密关联，分开输出会变成逐文件罗列。

以下情况不要合并：

- 只是名称相似，但代码目录、职责、入口或数据对象不同。
- 一个模块只是调用另一个模块，且二者有清晰独立职责。
- 仅共享公共库、工具类、框架代码或基础组件。

最终模块名称使用业务或功能名称，不使用普通文件名、普通类名、普通函数名。

## 字段归并规则

- `代码目录`：合并所有主要相对目录或文件路径，去重；一个路径时可输出字符串，多个路径时输出字符串数组。
- `面临威胁`：合并候选威胁并去重，保留与最终模块职责直接相关的威胁。
- 6 个“是否”字段：只要候选证据能够支持该特征，最终填 `是`；缺少证据支持时填 `否`。
- `是否外部暴露面`：只有存在对外接口、外部输入、协议入口、消息入口、文件上传、远程命令或外部输入主要处理环节的证据时填 `是`，否则填 `否`。
- `判断为高风险模块的原因`：重新整理为一段清楚说明，必须覆盖代码目录、模块职责、命中的高风险特征、外部暴露判断、合并理由和主要威胁依据。

不得虚构候选中没有依据的模块、目录、接口、调用关系或外部暴露面。

## 输出要求

仅输出合法 JSON 数组，不得输出 Markdown、解释文字、注释或代码块。

每个对象必须且仅包含以下 10 个字段：

1. `模块名称`
2. `代码目录`
3. `面临威胁`
4. `是否涉及设备或系统对外提供管理和控制接口相关的代码`
5. `是否涉及对不可信来源数据进行解析或处理的代码`
6. `是否涉及安全相关类代码(如，认证、授权、接入控制、加解密、密钥管理、日志审计、软件完整性保护等模块)`
7. `是否涉及个人数据或者敏感数据的代码`
8. `是否涉及web相关处理`
9. `是否外部暴露面`
10. `判断为高风险模块的原因`

所有“是否”字段的值只能是字符串 `是` 或 `否`。

没有高风险模块时输出 `[]`。

## 输出示例

```json
[
  {
    "模块名称": "用户认证模块",
    "代码目录": [
      "src/auth",
      "src/user/session"
    ],
    "面临威胁": "身份伪造、认证绕过、凭据泄露",
    "是否涉及设备或系统对外提供管理和控制接口相关的代码": "否",
    "是否涉及对不可信来源数据进行解析或处理的代码": "是",
    "是否涉及安全相关类代码(如，认证、授权、接入控制、加解密、密钥管理、日志审计、软件完整性保护等模块)": "是",
    "是否涉及个人数据或者敏感数据的代码": "是",
    "是否涉及web相关处理": "是",
    "是否外部暴露面": "是",
    "判断为高风险模块的原因": "src/auth 和 src/user/session 中的登录、令牌签发、会话校验和用户凭据处理共同构成用户认证模块；这些代码处理外部 HTTP 登录请求和用户凭据，命中不可信来源数据处理、web 处理、安全相关代码和敏感数据处理。登录路由对外提供访问入口，因此属于外部暴露面。多个候选分别描述认证入口和会话处理层，职责紧密关联，应合并为同一业务模块。"
  }
]
```
