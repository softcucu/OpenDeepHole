# 价值资产识别

## 任务目标

分析代码仓库中的产品文档、源代码、配置、接口、数据模型、部署文件和软件产物，从攻击者视角识别产品中的价值资产。

价值资产是攻击者为了实现攻击目的，可能获取、使用、控制、篡改、破坏或滥用的对象。

价值资产有且仅有以下四类：

- 数据资产
- 软件资产
- 硬件资产
- 服务资产

不得输出其他资产类别。

## 识别方法

以攻击意图为起点，反向识别攻击者可能攻击的资产。

依次分析以下攻击意图：

1. 获取系统资源或服务，如滥用计算、存储、网络、消息发送、AI推理、License或业务能力。
2. 窃取价值数据，如个人数据、业务数据、账号口令、Token、密钥、证书、配置、日志、模型和备份数据。
3. 阻碍服务正常提供或破坏网络、设备，如拒绝服务、异常报文、资源耗尽、配置破坏和关键数据删除。
4. 劫持设备并发起跳板攻击，如控制服务器、终端或网络设备，执行命令、安装软件、升级固件或横向移动。
5. 窃取设备传输内容，如通信报文、文件、消息、音视频、请求响应和同步数据。

对每种攻击意图，判断：

- 攻击者希望实现什么目的；
- 实现该目的需要攻击什么对象；
- 该对象属于哪一种价值资产；
- 代码仓库中是否存在对应依据；
- 资产被泄露、篡改、破坏、劫持或滥用后会造成什么损失。

## 资产分类

### 数据资产

系统存储、处理、生成、传输或管理的数据，例如：

- 个人数据和用户数据
- 业务数据
- 账号、口令、Token、密钥和证书
- 权限和角色数据
- 配置、日志、告警和License数据
- 模型、策略、备份和元数据
- 通信报文、文件、消息和音视频内容

### 软件资产

产品包含、发布、安装、升级、运行或依赖的软件及软件产物，例如：

- 应用程序
- 安装包、升级包和补丁包
- 插件、组件、SDK和工具
- 操作系统镜像和容器镜像
- 数据库、中间件、驱动和固件
- 关键脚本和算法实现

不得将普通类、函数、源代码文件或工具类单独识别为软件资产。

### 硬件资产

产品运行、管理、控制或依赖的物理设备和硬件资源，例如：

- 服务器和存储设备
- 网络设备和安全设备
- 终端设备和计算设备
- 芯片、硬盘和存储介质
- 串口、USB、蓝牙及其他硬件接口
- 被产品管理的外部设备

可通过驱动代码、固件代码、设备管理代码、设备通信协议和硬件控制接口识别硬件资产。

### 服务资产

系统对外或对内提供的业务、管理、运维、监控或资源能力，例如：

- 核心业务服务
- 认证、用户和权限管理服务
- 设备、配置和远程运维服务
- 告警、监控和日志审计服务
- 文件、消息和数据同步服务
- 软件升级和任务调度服务
- 计算、存储、代理和AI推理服务

不得将单个API、Controller、端口或函数直接作为服务资产，应归纳为完整的业务或系统服务。

## 攻击损失描述要求

攻击损失应描述资产遭受攻击后对产品、用户、业务或网络造成的直接后果，重点考虑：

- 数据泄露、篡改、删除或被越权使用；
- 账号冒用、权限提升或系统被非法控制；
- 服务中断、设备脱管或业务无法正常提供；
- 软件被篡改、植入恶意代码或供应链被破坏；
- 硬件资源被滥用、设备被劫持或成为跳板；
- 通信内容、商业信息或机密信息泄露；
- 引发隐私、合规、声誉、商业或网络安全风险。

攻击损失应结合资产用途进行描述，不得使用与资产无关的通用风险表述。

## 分析要求

- 优先阅读README、产品说明、架构文档、接口文档和部署文档，理解产品用途及核心能力。
- 结合数据模型、数据库脚本、接口、配置、网络通信、设备管理、构建和发布文件进行识别。
- 不得仅依据文件名或目录名判断资产。
- 不得将攻击入口、安全措施或普通代码模块直接作为价值资产。
- 登录接口、上传接口和管理端口等攻击入口，应识别其背后的认证服务、存储服务或设备管理服务。
- 输入校验、限流和访问控制等属于安全措施，不作为价值资产。
- 同一资产在多个模块中出现时应合并。
- 使用业务化资产名称，不得直接使用类名、函数名或文件名。
- 不得虚构代码仓库中不存在的资产。
- 没有仓库或产品文档依据的候选资产不输出。

## 判断原因要求

`判断为价值资产的原因` 必须结合代码分析证据详细说明为什么该对象是价值资产。

该字段应至少说明：

- 代码仓库中哪些文件、模块、接口、数据模型、配置、部署文件或软件产物体现了该资产；
- 这些代码或文档如何表明系统在存储、处理、传输、管理、控制、发布、安装、运行或依赖该对象；
- 攻击者可能为了哪类攻击意图获取、使用、控制、篡改、破坏或滥用该对象；
- 为什么该对象应归入当前资产类别，而不是攻击入口、安全措施、普通代码模块或普通函数。

原因说明应具体到可核验的代码或文档依据。可以引用相对文件路径、模块名、接口名、数据表名、配置项、部署对象或软件产物名称。不得只写“代码中存在相关逻辑”“该资产很重要”等泛化表述。

## 输出要求

最终只输出JSON，不输出 JSON 之外的分析过程、攻击意图、攻击方式、风险等级或其他说明。

输出必须是合法JSON，禁止使用Markdown代码块包裹，禁止在JSON前后添加任何文字。

输出结构如下：

[
  {
    "资产名": "用户个人数据",
    "资产类别": "数据资产",
    "资产描述": "系统处理和保存的用户身份信息、联系方式及其他可识别用户的数据。",
    "攻击损失": "资产泄露、篡改或被越权使用可能导致个人隐私泄露，并引发合规、声誉和商业风险。",
    "判断为价值资产的原因": "代码中的用户资料模型、用户信息查询和更新接口体现系统会保存、处理并对外返回用户身份信息和联系方式；这些数据可被攻击者用于窃取价值数据、冒用账号或进行隐私侵害。该对象是系统管理和处理的数据集合，不是单个接口或安全措施，因此归类为数据资产。"
  },
  {
    "资产名": "设备管理服务",
    "资产类别": "服务资产",
    "资产描述": "系统用于查询设备状态、修改设备配置、下发任务和控制设备的服务。",
    "攻击损失": "服务被劫持或破坏可能导致设备脱管、配置被恶意修改、业务中断，或设备被用于发起跳板攻击。",
    "判断为价值资产的原因": "代码中的设备状态查询、配置修改、任务下发和远程控制相关接口共同体现系统提供完整设备管理能力；攻击者可通过控制或滥用该服务修改设备状态、破坏配置或发起跳板攻击。该对象是完整业务服务能力，不是单个管理接口或端口，因此归类为服务资产。"
  }
]

每个资产对象必须且仅包含以下字段：

- `资产名`
- `资产类别`
- `资产描述`
- `攻击损失`
- `判断为价值资产的原因`

字段要求：

- `资产名`：使用简洁、明确的业务化名称。
- `资产类别`：只能填写`数据资产`、`软件资产`、`硬件资产`或`服务资产`。
- `资产描述`：简要说明资产包含的内容和主要用途。
- `攻击损失`：简要说明资产被泄露、篡改、破坏、劫持或滥用后可能造成的后果。
- `判断为价值资产的原因`：结合代码分析证据详细说明该对象为什么属于价值资产，并说明其仓库依据、资产类别依据和攻击者可能滥用该对象的原因。

不得输出额外字段，不得输出空字段，不得输出注释，不得使用尾随逗号。
