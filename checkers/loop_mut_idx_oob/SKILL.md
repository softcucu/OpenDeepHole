# 循环变化索引越界审计

你正在核实一处 C/C++ 候选越界线索。候选形态可能是：某个索引变量 `$IDX` 在 `for` / `while` / `do-while` 循环中变化，同时被用于数组下标、指针偏移、结构体指针访问或内存函数参数；也可能是 `copy_from_user` 家族调用 `$COPY($DST, ..., $LEN)` 使用重点变量 `$LEN`，而目标 `$DST` 在循环内按 `$LEN` 累加或递减。

该线索只来自语法层面的初筛，而且故意偏宽以提高候选召回。你的任务不是确认“代码看起来可疑”，也不是因为循环条件没有直接写 `$IDX` 就确认漏洞，而是判断是否存在真实、可达的越界读写风险。只有能证明 `$IDX`、`$DST` 或 `$LEN` 的组合可能越过被访问对象的真实边界，并能说明漏洞如何被触发时，才确认漏洞。

## 可用工具

- `submit_result(confirmed, severity, description, ai_analysis)` - 审计完成后必须调用

## 审计流程

### Step 1 - 读取完整函数

先阅读候选所在函数的完整源码。不要只依据候选描述或上下文片段判断。

确认以下信息：

- `$IDX` 或 `$DST` 在哪里初始化、如何变化、是否可能回绕或跨步增长
- `$COND` 控制的真实循环次数是什么
- 如果候选给出重点变量 `$LEN`，确认它的来源、取值范围、是否来自用户/协议/结构体字段
- `$IDX` 被用于哪个数组、指针、结构体成员或内存函数参数
- `$COPY($DST, ..., $LEN)` 这类调用中 `$DST` 指向的真实对象和 `$LEN` 表示的真实拷贝长度
- 访问发生在索引变化之前还是之后
- 命中点前后是否有边界检查、fail-fast、assert、宏或 wrapper
- 由于候选偏宽召回，必须主动寻找误报证据，不能默认相信候选线索

### Step 2 - 还原真实边界

不要只看变量名。必须尽量还原被访问对象的真实边界：

- 局部数组：看声明大小
- 结构体成员数组：查看结构体定义
- 指针参数：追踪调用契约、长度参数、分配长度或上层校验
- 全局数组：查看全局定义和配套长度常量
- 指针偏移：计算偏移后剩余空间，而不是完整对象大小
- 内存函数：确认参数中 `ptr + idx` 或 `&arr[idx]` 对应的真实可写/可读范围
- `copy_from_user` 家族：确认 `$DST` 的目标缓冲区总容量、循环前剩余空间、每轮 `$LEN` 和累计 `sum($LEN)` 是否可能超过容量

如果无法证明对象边界，也无法证明 `$IDX` 或 `$DST + $LEN` 会越界，优先判为误报，并说明缺失的关键事实。

### Step 3 - 判断循环条件、长度变量与目标边界

候选线索只识别了一些直接条件和局部比较，无法理解等价约束。重点检查：

- `$COND` 是否约束了与 `$IDX` 同步变化的变量，例如 `remain`、`byteNum`、`contentLen`、`offset`
- `$IDX` 是否由固定步长从 0 开始，并且循环次数由真实数组容量派生
- `$COND` 是否来自同一个长度字段或分配长度
- `$LEN` 是否来自外部输入、协议字段、用户参数或未可信的结构体字段
- `$LEN` 是否在本函数循环前、循环内、调用方或解析阶段已被限制到 `$DST` 剩余空间以内
- `$DST` 是否每轮按 `$LEN` 推进，推进后的下一轮写入是否仍在同一目标对象内
- 循环体内是否在访问前更新了 `$IDX`，导致边界关系被破坏
- 是否存在 `break`、`return`、`goto cleanup` 等提前退出路径

如果能证明 `$COND`、`$LEN` 校验或 `$DST` 剩余空间计算间接保证访问不越界，判为误报。

宽召回可能命中这些安全写法，必须逐一排除：

- `remain`、`len` 与 `$IDX` 同步变化，且初值已经被上游限制在目标容量内
- 循环进入前已有 `len <= sizeof(buf)`、`count <= cap`、协议字段合法性检查
- 每轮 `copy_from_user` 前已有 `len <= remaining`、`offset + len <= cap` 或等价检查
- `$IDX` 虽然不在 `$COND` 里，但 `$COND` 使用的变量本身就是目标数组剩余容量
- `$IDX` 的变化方向和访问顺序保证访问范围更小，例如先检查/先递减再访问

### Step 4 - 还原漏洞触发方式

如果倾向确认漏洞，必须说明可触发方式，而不是只给出“可能越界”：

- 哪些输入或结构体字段控制循环次数、`$LEN`、用户态源地址或目标对象大小
- 攻击者需要让 `$LEN` 取什么范围，或让多轮 `$LEN` 累计达到什么关系
- 第几轮或在什么条件下 `$DST + $LEN` 首次越过真实边界
- `copy_from_user` 失败返回是否会阻断越界；如果越界写已在返回前发生，需要明确说明
- 如果缺少调用链或目标容量事实，不能编造触发方式，应判为误报或信息不足
- 为了稳定触发漏洞，可以设置越界容量为一个数据类型的最大值，比如0XFFFFFF

### Step 5 - 检查局部防护

初筛已过滤了一些明显 `if (idx < bound)` 和 fail-fast 形态，但仍可能漏掉宏或封装。必须继续检查：

- `if` / `switch` / 三元表达式中的范围校验
- `ASSERT`、`CHECK`、`RETURN_IF`、`goto fail`、`break`、`continue`
- 项目内校验宏或 wrapper，例如 `CHECK_INDEX(idx, len)`
- 上游解析阶段是否已经验证协议长度、包长度、数组容量
- 访问前是否 clamp 了 `$IDX`
- `copy_from_user` 前是否校验 `$LEN`、剩余空间、累计 offset 或目标 buffer 总长度

只有边界检查不能覆盖当前访问路径时，才继续考虑确认。

### Step 6 - 作最终判定

确认真实漏洞需要同时满足：

- `$IDX` 或 `$DST` 在循环中变化，且可达路径上可能超过目标对象的有效范围
- 目标对象的真实大小或剩余空间小于 `$IDX`、`$DST` 偏移或 `$LEN` 累计可能达到的位置
- 循环条件、长度校验或剩余空间计算没有直接或间接提供等价约束
- 访问点前没有覆盖该路径的范围检查、fail-fast 或断言
- 越界访问会真实发生，而不是只导致安全函数失败或不可达分支
- 能描述具体触发方式，至少包括外部可控输入、边界关系和到达命中 sink 的路径
- 只有确认是越界问题，才确认是真实漏洞，其它类型问题忽略

以下情况判为误报：

- `$COND` 虽不含 `$IDX`，但通过同步变量、长度字段或固定循环次数间接约束了 `$IDX`
- 访问对象容量由协议、分配长度、数组常量或调用方契约保证
- 命中点前已有有效校验，只是初筛没识别宏、wrapper 或复杂条件
- `$IDX` 的最大值小于数组容量，或访问表达式/`$LEN` 累计后的偏移仍在剩余空间内
- 代码位于测试、mock、stub 或不可达分支
- 信息不足，无法证明真实越界路径或漏洞触发方式

## 严重程度

- `high`：外部输入、网络包、文件内容或用户请求可控制循环次数/长度/索引，并能触发越界读写。
- `medium`：真实可达，但主要来自内部状态、错误路径或攻击者控制较弱的路径。
- `low`：确认存在越界，但触发条件苛刻、影响范围小或仅低频路径。

## 提交结果

分析完成后必须调用 `submit_result`：

- `confirmed`：只有确认真实越界时为 `true`
- `severity`：`high` / `medium` / `low`
- `description`：一句话说明真实问题或误报原因
- `ai_analysis`：写清 `$IDX`/`$DST`/`$LEN` 的变化、循环条件、目标对象边界、已有校验、可达性、漏洞触发方式和最终判断
