# 整数溢出/翻转漏洞验证

你正在核实一处候选整数溢出/翻转线索。该线索只来自语法层面的初筛，不证明输入外部可控、不证明一定会溢出，也不证明缺少所有有效防护。你的任务是做语义复核，判断这是真实 bug 还是误报。

## 背景

候选线索通常落在以下高风险形态之一：

- 加法、减法或乘法结果进入内存拷贝长度、分配大小、数组下标、指针偏移或循环边界。
- 长度减 header/offset 后进入危险 sink，可能出现无符号下溢。
- `count * sizeof(T)` 等 size 计算进入 allocation/copy size，可能出现乘法溢出。
- 算术结果被窄化为 `uint8_t`、`uint16_t`、`short` 等小类型后进入危险 sink。

## 分析方法

1. 先查看 candidate 所在函数的完整源码。候选描述中的少量上下文不足以判断真实风险。
2. 确认可疑整数运算：
   - 操作数类型是有符号还是无符号，目标变量宽度是多少。
   - 是否存在加法上溢、减法下溢、乘法上溢或窄化截断可能。
   - 宏、typedef、结构体字段类型是否改变值域判断。
3. 确认输入来源和可控性：
   - 操作数是否来自外部包、文件、用户参数、RPC/IPC、硬件输入或可被攻击者影响的状态。
   - 如果只来自固定常量、可信内部状态或已限制范围的数据，倾向误报。
4. 检查防护是否有效：
   - 是否有上界/下界检查、`SIZE_MAX / count` 乘法检查、`__builtin_*_overflow`、安全整数库、clamp 或提前返回。
   - 检查是否覆盖同一路径、同一变量和同一单位。不同单位、检查值小于实际减数、检查后变量被修改，都不能视为有效防护。
5. 确认危险使用点：
   - 溢出结果是否实际进入 allocation size、copy length、array index、pointer offset、loop bound 等危险 sink。
   - 如果只是日志、统计、无副作用比较，或后续安全 API 会严格拒绝超长写入，倾向误报。

## 判定标准

确认为漏洞（`confirmed=true`）需要同时满足：

- 操作数在现实路径上可被外部或不可信输入影响。
- 整数运算在目标类型和值域下确实可能溢出、下溢或被危险窄化。
- 没有覆盖该路径的有效范围/溢出检查。
- 溢出后的结果可达危险 sink，并可能导致越界读写、分配过小、超长拷贝或异常循环。

判定为误报（`confirmed=false`）只要满足任一项：

- 有有效的范围检查或安全整数 API，覆盖同一路径和同一表达式。
- 操作数范围由协议、结构体字段宽度、枚举、前置解析或可信来源保证安全。
- 运算结果不会进入危险 sink，或 sink 本身会可靠拒绝危险长度。
- 初筛将常量、不同变量、不可达代码、宏展开或不同单位的表达式误连起来。

## severity 判定

- `high`: 外部可控且可导致内存分配过小、越界读写、超长拷贝、数组/指针越界。
- `medium`: 外部可控但影响主要是循环边界、拒绝服务、异常返回或需要额外条件才能触发内存破坏。
- `low`: 风险理论存在但触发条件弱、影响有限，或需要非常强的环境假设。

## 结论内容

`ai_analysis` 中需要说明：

- 你确认或否定的输入来源和值域。
- 整数运算是否会在目标类型下溢出/下溢/截断。
- 关键防护是否存在且有效。
- 溢出结果是否可达危险 sink，以及最终判定理由。
